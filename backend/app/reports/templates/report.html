{# Remediation Roadmap #}

Remediation Roadmap

Priority 1 — Rotate all verified-active credentials immediately Any finding marked Verified Active represents a live, exploitable credential. Rotate it in the relevant service console (AWS IAM, Stripe dashboard, GitHub developer settings) before taking any other action.

Priority 2 — Remove secrets from the current branch Delete the secret from the file and commit the change. This prevents future exposure but does not remove it from git history.

Priority 3 — Clean git history Use BFG Repo-Cleaner (java -jar bfg.jar --replace-text secrets.txt) or git filter-repo --replace-text secrets.txt to purge the secret from all commits. Force-push the cleaned history and notify all collaborators to re-clone.

Priority 4 — Implement preventive controls Install a pre-commit secret scanner (TruffleHog or GitLeaks) to catch secrets before they are committed. Store all credentials in environment variables or a dedicated secrets manager (AWS Secrets Manager, HashiCorp Vault, or similar).

Appendix: Methodology & Tool Versions

Scan performed by Redact using TruffleHog as the scanning backend. TruffleHog scans full git history across all branches (--all-branches) and performs live credential verification for all supported detector types. Findings with Verified: true were confirmed active via live API calls at the time of scan. The --only-verified flag was not used — all findings (verified and unverified) are included.

Compliance mappings reference NIST SP 800-53 Rev 5 and the DISA Application Security and Development STIG. Control applicability is determined by secret type as detected by TruffleHog.
